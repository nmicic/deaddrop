# SPEC DRAFT F — Self-hosted VM Variant

> **Status: GRADUATED — frozen historical snapshot.** F has graduated
> out of `experimental/` and become the `vm-tx` backend profile. See
> `../BACKEND_VM.md` and `../DECISIONS.md` D-24. Do not edit this file;
> edits belong in `BACKEND_VM.md`. Kept only so old cross-references and
> design notes remain valid.
>
> **Superseded storage choice:** this snapshot describes a bbolt
> on-disk transactional store. D-39 replaces that with an in-memory
> mlocked map — see `../BACKEND_VM.md §3` and D-39 for the current
> design. The bbolt wording below is historical only.

Same wire protocol (`PROTOCOL.md`), different relay implementation. Trades
Cloudflare's global edge for a VM you own.

Reuses a hardened self-hosted VM security posture.

---

## When to choose F

- You want larger blobs than Cloudflare KV's 25 MiB value limit.
- You do not want a Cloudflare dependency.
- You want operational control (logs, rate limits, firewall, private network).
- You already operate a VM and can add one more service cheaply.
- You want to run entirely on LAN or Tailscale (no public exposure).

## When NOT to choose F

- You want zero-ops — use the Cloudflare Worker variant.
- You do not have a VM and do not want to run one.
- You want anycast / global edge — a single VM cannot provide it.

## Reference stack

```
           internet
              │
         ┌────┴────┐
         │  caddy  │   TLS 1.2+, Let's Encrypt, optional mTLS (variant D)
         └────┬────┘
              │ unix domain socket  /run/deaddrop/app.sock
         ┌────┴────┐
         │deaddrop │   Go, single static binary
         │ -server │
         └────┬────┘
              │
         ┌────┴────┐
         │  bbolt  │   on-disk key-value store, ≤10 MiB rows
         │ (encrypt│   on encrypted volume (LUKS / FileVault / APFS)
         │ volume) │
         └─────────┘
```

Storage schema (bbolt bucket `slots`):

```
key   = slot_id (hex, 16 bytes)
value = protobuf { ciphertext, expires_at_unix, reads_left }
```

Background goroutine sweeps expired rows every 60s (a standard
Go service's TTL-cleanup pattern).

## Differences from Cloudflare variant

```
+  Configurable MAX_BLOB_BYTES up to disk capacity.
+  Standard ops tooling (journald, nftables/iptables, fail2ban).
+  Can run on LAN / Tailscale — zero public exposure option.
+  Structured logs with opaque slot hashes (no content leak).
+  Per-IP rate limits via caddy / nginx.
-  You run it. Patching, monitoring, backups are on you.
-  No global anycast; receiver latency depends on VM location.
-  Must rotate Let's Encrypt certs (caddy does this automatically).
```

## Security posture

See `../SECURITY.md` and `../BACKEND_VM.md` — the layer-by-layer attacker walk.
Applied to deaddrop:

- **Firewall inbound** (nftables): 443 + 22 only. SSH rate-limited to
  5 new conns/min. HTTPS rate-limited to 200 conns/s with burst 50.
- **Firewall outbound**: whitelist 53/80/443 only. Prevents C2 exfil if the
  deaddrop binary is ever RCE'd.
- **TLS**: 1.2+ only, HSTS, no weak ciphers.
- **Unix-socket between caddy and server**: no TCP listener for the Go app.
  Nothing else on the host can reach it without filesystem permission.
- **Systemd hardening**: `PrivateTmp`, `ProtectSystem=strict`,
  `NoNewPrivileges`, `CapabilityBoundingSet=`, dedicated `deaddrop` user.
- **Optional Cloudflare tunnel** (`cloudflared`): if you want edge + your VM,
  expose through Cloudflare without opening a public port.

## Single-user knobs (the "just me" deployment)

```
MAX_BLOB_BYTES    = 10 MiB
MAX_TTL           = 600 s   (10 min)
MAX_READS         = 1       (strict one-shot; raise only when distributing)
WRITE_TOKEN       = enabled
ALLOWED_IPs       = (optional) Tailscale ACL only
```

## Build and deploy

```
make build                                # produces ./deaddrop-server

./scripts/install-vm.sh user@vm.example.com
  # - copies binary to /usr/local/bin/deaddrop-server
  # - drops /etc/systemd/system/deaddrop.service
  # - creates deaddrop user, /var/lib/deaddrop, /run/deaddrop
  # - renders /etc/caddy/Caddyfile fragment
  # - runs: systemctl enable --now deaddrop && systemctl reload caddy
```

Secrets (not in image):

```
/etc/deaddrop/deploy_secret      # mode 0600, owned by deaddrop
/etc/deaddrop/write_token        # mode 0600
/etc/deaddrop/ca.pem             # (variant D) CA bundle for mTLS
```

## Compatibility guarantee

A client built for variant A/B/C works against either a Cloudflare Worker
relay or a VM relay without code changes. The only client-side difference is
`RELAY_BASE_URL`. This is enforced by the single-protocol invariant in
`ARCHITECTURAL_PRINCIPLE_GENERIC_PIPELINE.md`.
